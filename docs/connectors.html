Connector types

All SteroidDoseR functions accept a connector object as their first argument. All connectors implement the same interface, so the same calling code works unchanged against a live database or an in-memory data frame.

ConstructorUse case
create_df_connector(df)In-memory data frame, synthetic data, unit tests — no database packages required
create_omop_connection(...)Live OMOP CDM — high-level builder, reads config from env vars. Recommended for most sites.
create_connection_from_env(".env")Load a .env file and open a connection in one call — simplest setup
create_omop_connector(connectionDetails, ...)Low-level — supply a pre-built connectionDetails object from DatabaseConnector

Functions also accept a plain data frame directly (legacy path). When a data frame is supplied, connector-path arguments (drug_concept_ids, person_ids, start_date, end_date, sig_source) are silently ignored.

Live OMOP database

Prerequisites

  • DatabaseConnector and SqlRender installed
  • Read-only access to a schema containing drug_exposure, concept, and drug_strength
  • JDBC driver files for your database platform

JDBC driver setup

# Download drivers once to a local folder
DatabaseConnector::downloadJdbcDrivers("sql server", pathToDriver = "~/jdbc")
DatabaseConnector::downloadJdbcDrivers("postgresql", pathToDriver = "~/jdbc")
# other options: "redshift", "oracle", "bigquery", "spark", "snowflake"

# Add to ~/.Renviron so the path is never hard-coded in scripts:
# JDBC_DRIVER_PATH=/path/to/jdbc

Using a .env file (recommended)

con      <- create_connection_from_env(".env")
episodes <- run_pipeline(con, method = "baseline")
disconnect_connector(con)

SQL Server — Windows integrated auth

con <- create_omop_connection(
  dbms             = "sql server",
  server           = "myserver.institution.edu",
  database         = "OMOP_CDM",
  use_windows_auth = TRUE,
  cdm_schema       = "MyDatabase.dbo"
)

SQL Server — username / password

con <- create_omop_connection(
  dbms       = "sql server",
  server     = Sys.getenv("SQL_SERVER"),
  database   = Sys.getenv("SQL_DATABASE"),
  user       = Sys.getenv("SQL_USER"),
  password   = Sys.getenv("SQL_PASSWORD"),
  cdm_schema = Sys.getenv("SQL_CDM_SCHEMA")
)

PostgreSQL

con <- create_omop_connection(
  dbms       = "postgresql",
  server     = "localhost",
  database   = "omop_cdm",
  user       = Sys.getenv("DB_USER"),
  password   = Sys.getenv("DB_PASSWORD"),
  port       = 5432,
  cdm_schema = "cdm_54"
)

Databricks / Spark

con <- create_omop_connection(
  dbms          = "databricks",
  server        = "workspace.cloud.databricks.com",
  database      = "default",
  user          = "token",
  password      = Sys.getenv("DATABRICKS_TOKEN"),
  cdm_schema    = "omop.data",
  extraSettings = "httpPath=/sql/1.0/warehouses/<warehouse-id>"
)

Separate vocabulary schema v0.1.6 Many OMOP sites store vocabulary tables (concept, drug_strength) in a different schema from drug_exposure. Pass vocab_schema = "my_vocab_schema" to create_omop_connector() — it is now wired through to the SQL template. When omitted, vocab_schema defaults to cdm_schema. 
con <- create_omop_connector(
  connectionDetails,
  cdm_schema   = "my_cdm",
  vocab_schema = "my_vocab"   # vocabulary tables in a separate schema
)

Capability detection

OMOP implementations vary by site. detect_capabilities() probes which drug_exposure columns are populated and stores the results on the connector so imputation functions can adapt automatically. 

con <- detect_capabilities(con)
con$capabilities
#> $has_sig         TRUE
#> $has_days_supply TRUE
#> $has_quantity    TRUE
#> $has_route       TRUE

Sites without a SIG column

If drug_exposure.sig is absent, pass sig_source = "drug_source_value" to use the source string instead. The package will also insert a blank sig column automatically (all rows return parsed_status = "empty") and rely on the strength fallback from amount_value. 

doses <- calc_daily_dose_nlp(con, sig_source = "drug_source_value")

Filtering by patient or drug concept

# Restrict to specific patients at the SQL level (cohort filter)
doses <- calc_daily_dose_baseline(con, person_ids = c(1001L, 1002L))

# Restrict to specific OMOP drug concept IDs
doses <- calc_daily_dose_nlp(con, drug_concept_ids = c(1518254L, 40224131L))

Column contract

When supplying a plain data frame, columns must follow the OMOP drug_exposure schema. See ?drug_df_contract for the full specification.

ColumnTypeRequired byNotes
person_idint/chrAllPatient identifier
drug_exposure_start_dateDateAllExposure start
drug_exposure_end_dateDateBaseline M4, episodesOptional — today substituted if absent (with warning)
drug_concept_namechrNLP, Baseline fallbackDrug name for standardisation and mg extraction
drug_source_valuechrBaseline strength fallbackSource drug string; mg extracted via regex
amount_valuenumBaseline M2–M4, NLP fallbackPer-tablet strength from drug_strength JOIN
amount_unit_concept_idintBaseline unit checkOMOP concept for unit; 8576 = mg, 0 = unmapped (treated as unknown). Non-mg units are discarded.
quantitynumBaseline M3, M4Tablets dispensed
days_supplynumBaseline M3Days of supply
sigchrNLPFree-text SIG instruction
route_concept_namechrNLP oral filterRoute standardised name; used to restrict to oral records
route_concept_idintNLP oral filter fallbackUsed when route_concept_name is absent
