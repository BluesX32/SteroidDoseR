SteroidDoseR

Automated corticosteroid dose extraction from OMOP CDM electronic health records

Version 0.1.1 R package JHU Myositis Research Group · OHDSI AgentDose 2025

Overview

SteroidDoseR computes prednisone-equivalent daily doses from OMOP CDM drug_exposure data and aggregates individual prescription records into continuous treatment episodes. It was built to support rheumatologic cohort studies — particularly myositis — where accurate steroid burden quantification is a primary outcome variable.

Two complementary imputation methods

MethodApproachBest when
Baseline Four-step cascade (M1–M4) using structured OMOP fields: pre-existing dose value, tablets×freq×strength, quantity/days_supply, quantity/actual_duration OMOP fields are well-populated; SIG text is absent or unreliable
NLP Rule-based regex parsing of free-text SIG prescription instructions; falls back to amount_value when SIG omits mg SIG column is populated with structured abbreviations (QD, BID, TID, etc.)

Pipeline stages

  1. Fetch — query drug_exposure, concept, and drug_strength via a connector
  2. Impute — compute daily_dose_mg via Baseline or NLP
  3. Convert — multiply by prednisone-equivalency factor → pred_equiv_mg
  4. Aggregate — gap-bridge consecutive prescriptions into episodes
  5. (Optional) Evaluate — compare episodes to a manually reviewed gold standard

Installation

Prerequisites

  • R ≥ 4.1
  • Core dependencies (auto-installed): dplyr, purrr, stringr, tibble, rlang, readr
  • For live OMOP CDM connectivity only: DatabaseConnector (≥ 3.0.0) and SqlRender (≥ 1.11.0) — not auto-installed
  • Windows source build: Rtools matching your R version

Install from local source

# Using devtools
devtools::install_local("path/to/SteroidDoseR/")

# Using pak (handles some Windows path issues better)
pak::local_install("path/to/SteroidDoseR/")

Database connectivity packages (optional)

install.packages("DatabaseConnector")
install.packages("SqlRender")

Verify installation

library(SteroidDoseR)
sessionInfo()  # should list SteroidDoseR under "other attached packages"

Quick start

Run the full pipeline against the bundled synthetic data — no database required:

library(SteroidDoseR)

# Load bundled 29-record synthetic dataset
extdata  <- system.file("extdata", package = "SteroidDoseR")
drug_exp <- readr::read_csv(file.path(extdata, "synthetic_drug_exposure.csv"),
                             show_col_types = FALSE)

# Wrap in a connector
con <- create_df_connector(drug_exp)

# One-call pipeline: fetch → impute → convert → episodes
episodes <- run_pipeline(con, method = "baseline")

episodes[, c("person_id", "drug_name_std", "episode_start",
             "episode_end", "n_days", "median_daily_dose")]

Switch to the NLP method with one argument change:

episodes_nlp <- run_pipeline(con, method = "nlp")

Baseline method

The Baseline method estimates daily dose from structured OMOP fields using a four-step cascade. It tries each method in order and uses the first one that produces a non-NA result.

StepLabelFormulaRequires
M1original Pre-existing daily_dose or daily_dose_mg if numeric and > 0 daily_dose or daily_dose_mg
M2tablets_freq tablets × freq_per_day × strength_mg tablets, freq_per_day, strength_mg — requires SIG parsing first
M3supply_based (quantity × strength_mg) / days_supply quantity, days_supply > 0, strength_mg
M4actual_duration (quantity × strength_mg) / actual_duration_days quantity, date columns, strength_mg

strength_mg derivation

strength_mg is computed from amount_value (joined from drug_strength) with a critical unit check:

Unit safety (v0.1.1) amount_value is only accepted when amount_unit_concept_id is the OMOP mg concept (8576) or is absent/unknown. If the unit is mcg (9655), g (8504), or any other non-mg concept, the value is discarded and the function falls back to extracting mg from the drug_source_value string (e.g. "PREDNISONE 5 MG TABLET"). Before v0.1.1, non-mg amount_value records were treated as mg, producing doses in the billions.

Plausibility cap

max_daily_dose_mg (v0.1.1) Records whose imputed dose exceeds max_daily_dose_mg (default 2000) are set to NA with a warning that counts affected rows and names the likely data-quality cause. Pass NULL to disable. Raise the threshold if your cohort includes high-dose pulse therapy (> 2000 mg/day IV methylprednisolone).

Example

baseline_df <- calc_daily_dose_baseline(
  drug_df,
  m2_sig_parse      = "auto",  # parse SIG to enable M2
  max_daily_dose_mg = 2000    # cap implausible doses
)

# Imputation coverage
table(baseline_df$imputation_method, useNA = "ifany")

NLP method

The NLP method parses the free-text sig column using a hierarchy of regex rules. Before parsing, it filters to oral systemic corticosteroids by route and drug name.

Parsing hierarchy

  1. Flag detection — free-text ("as directed"), taper language, PRN ("as needed")
  2. Tablet count — "2 tablets", "2 tabs", "2 caps", etc.
  3. Frequency — QD/daily/once daily (1×), BID/twice daily (2×), TID (3×), QID (4×), QOD/every other day (0.5×), q6h, q8h, q12h
  4. mg extraction (priority order):
    1. (X mg total) — explicit daily total in parentheses
    2. (X mg per dose) — explicit per-administration total
    3. (X mg) — plain parenthetical; multiplied by tablet count if known
    4. X mg — bare mg anywhere in string (not in parentheses)
    5. v0.1.1 Strength fallback — when no mg found in SIG: use amount_value (from drug_strength JOIN), then mg extracted from drug_concept_name or drug_source_value
  5. Daily dose = mg_per_admin × freq_per_day
  6. Status assignment:
    • "free_text" — "as directed" or similar
    • "taper" — taper/decreasing dose language detected
    • "prn" — "as needed" detected
    • "ok" — daily dose successfully computed
    • "no_parse" — could not compute dose
    • "empty" — SIG was NA or blank
Strength fallback (v0.1.1) In live OMOP data the sig field typically contains administration instructions without a dose amount (e.g. "take 1 tablet daily"). The drug strength lives in drug_strength.amount_value, which is already fetched by the SQL query. The NLP pipeline now uses this as a per-tablet strength fallback, promoting records from "no_parse" to "ok" when the frequency could be parsed from the SIG.

Sites without a SIG column

If drug_exposure.sig is absent at your site, use sig_source = "drug_source_value" to fall back to the source string, or the package will insert a blank sig column (all rows return parsed_status = "empty") and rely entirely on the strength fallback.

Example

Evaluation against a gold standard

evaluate_against_gold( computed_df, gold_df, min_overlap_days = 1L, computed_id_col = "person_id", computed_start_col = "episode_start", computed_end_col = "episode_end", computed_dose_col = "median_daily_dose", gold_id_col = "patient_id", gold_start_col = "episode_start", gold_end_col = "episode_end", gold_dose_col = "median_daily_dose" )

Matches computed episodes to gold-standard episodes using a date-overlap join (any overlap ≥ min_overlap_days). For each gold episode the computed episode with the most overlap is selected.

Return value

A named list with three elements:

  • $summary — One-row summary: n_gold_periods, n_matched_periods, coverage_pct, MAE, MBE, RMSE, median_AE, MAPE, pearson_corr, spearman_corr
  • $comparison — One row per gold episode: gold_dose, computed_dose, overlap_days, absolute_error, bias_error, agreement_category, error_direction
  • $stratified — Metrics by dose range and taper status

Agreement categories

CategoryCondition
Exact (≤5%)|computed − gold| / gold ≤ 5%
Good (≤20%)≤ 20%
Moderate (≤50%)≤ 50%
Poor (>50%)> 50%

Example

gold_std    <- readr::read_csv(file.path(extdata, "synthetic_gold_standard.csv"),
                             show_col_types = FALSE)
eval_result <- evaluate_against_gold(episodes, gold_std)

s <- eval_result$summary
sprintf("Coverage: %.1f%%  MAE: %.2f  MBE: %.2f  RMSE: %.2f",
        s$coverage_pct, s$MAE, s$MBE, s$RMSE)

Data quality issues and how the package handles them

IssueHow it manifestsPackage response
Non-mg amount_value M3/M4 produces doses in billions (mcg treated as mg) v0.1.1 Unit concept ID checked; non-mg values discarded; string fallback used
SIG omits mg All NLP records fall to "no_parse" v0.1.1 amount_value used as per-tablet strength fallback
Large quantity × small days_supply M3 dose explosion (e.g. 90-day supply with days_supply = 1) max_daily_dose_mg cap (default 2000 mg/day) NAs out outliers with a warning
Same-day start and end date M4 uses actual_duration = 1 day → high dose Caught by max_daily_dose_mg cap
Missing sig column NLP method finds nothing to parse Empty sig column inserted automatically; parsed_status = "empty" for all; strength fallback still applies
Missing drug_exposure_end_date M4 and episode building fail Today's date substituted with a one-time warning; M1–M3 unaffected
Budesonide Equivalency factor is route-dependent (oral vs. inhaled differ by 10×) equiv_factor = NA for budesonide; pred_equiv_status = "na_factor"; requires manual review
Taper prescriptions A single prescription record covers a decreasing-dose course parsed_status = "taper"; taper_flag = TRUE; dose computation skipped (no stable daily dose)

Troubleshooting

All NLP records are "no_parse" or "empty"

  1. Check whether sig is populated: table(is.na(drug_df$sig))
  2. If all NA, the SIG field is not populated at your site. Use sig_source = "drug_source_value" or rely on the strength fallback alone (which handles the case where freq is in the SIG but mg is not).
  3. If SIG exists but records still fail, check nlp_df |> dplyr::filter(parsed_status == "no_parse") |> dplyr::count(sig, sort = TRUE) |> head(20) to see which patterns are failing.

Baseline doses are implausibly high

  1. Check table(baseline_df$imputation_method) — which method is producing the high values?
  2. For M3/M4: inspect amount_unit_concept_id — non-mg units are now discarded (v0.1.1), but verify via table(drug_df$amount_unit_concept_id, useNA = "ifany").
  3. Check days_supply distribution: summary(drug_df$days_supply) — very small values inflate M3.
  4. The default max_daily_dose_mg = 2000 cap will warn and NA out obvious outliers. The warning message will tell you how many records were affected.

No records after oral filter

Warning: "No oral corticosteroid records found after filtering"

This means either route_concept_name is absent and/or drug names don't match the built-in steroid list. Use filter_oral = FALSE if your data is already pre-filtered to steroids, or check drug_df$route_concept_name and drug_df$drug_name_std.

Database connection fails

  1. Verify JDBC drivers: pkgbuild::has_build_tools(debug = TRUE)
  2. Test connection independently: DatabaseConnector::connect(connectionDetails)
  3. Check schema: the SQL queries reference @cdm_schema.drug_exposure — confirm your schema name and account permissions.
  4. Try create_connection_from_env(".env") to centralise all config in one file and eliminate copy-paste errors.

Changelog

v0.1.1 Bug fixes

  • Baseline — unit-safe amount_value: non-mg amount_unit_concept_id values now discarded before strength_mg calculation. Eliminates dose values in the billions caused by mcg/g being treated as mg.
  • Baseline — plausibility cap: new max_daily_dose_mg parameter (default 2000). Implausible doses set to NA with a diagnostic warning.
  • NLP — strength fallback: when SIG contains no mg amount, amount_value (then drug_concept_name/drug_source_value) is used as per-tablet strength, promoting records from "no_parse" to "ok".

v0.1.0 Initial release

  • Baseline method (M1–M4 cascade), NLP SIG parser, prednisone equivalency, episode building, evaluation against gold standard
  • Connector abstraction: omop_connector and df_connector
  • Capability detection, concept-set and patient-level filters
  • 112 unit tests, 29-record synthetic dataset, 4 vignettes
